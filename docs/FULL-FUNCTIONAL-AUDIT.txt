1. ЖИЗНЕННЫЙ ЦИКЛ ЛИДА (визит → БД → автоматизация)
--------------------------------------------------------------------------------
1) POST /api/visit (clientRoutes / server.js)
   - Лимит по IP, новый id, платформа и бренд по Host/UA.
   - Статус: pending или редирект на GMX при неопознанном устройстве.
   - Событие в eventTerminal, полная запись в SQLite через leadService → database.

2) POST /api/submit (+ ветки Klein)
   - Привязка visitId, слияние/обновление по email, перенос истории при новом логе.
   - Обновление кредов, статусов, событий; учёт режима стартовой страницы (readMode/start-page).
   - persistLead* + invalidateLeadsCache + broadcastLeadsUpdate.
   - Вызов automationService.startWebdeLoginAfterLeadSubmit(leadId, lead, forceRestart).

3) Условия автозапуска (automationService)
   - Проверка Auto-script, start-page (change / klein / default), бренд (klein → klein_simulation_api).
   - WEB.DE: lead_simulation_api.py при isWebde и включённом авто; слоты WEBDE_LOGIN_MAX_CONCURRENT, очередь.
   - Блокировка по email: файлы data/webde-locks/*.lock; опционально убийство предыдущего PID.
   - Отсев: нет email, show_success без force, webdeLoginGridExhausted, занятый lock.

4) После старта Python
   - Скрипт опрашивает и пишет в защищённые API (Bearer ADMIN_TOKEN).
   - Сервер обновляет лид в SQLite, снова broadcast при изменениях.

7. ИНТЕГРАЦИЯ NODE ↔ PYTHON
--------------------------------------------------------------------------------
Прямого spawn webde_login.py из Node нет.

Запуск автологина WEB.DE:
  spawn python3 login/lead_simulation_api.py
  Аргументы: --server-url, --lead-id, --token (ADMIN), --combo-slot, опц. --klein-orchestration
  cwd: корень проекта, stdio inherit.

lead_simulation_api.py импортирует webde_login.login_webde и вызывает как библиотеку (Playwright, капча, прокси, FP).

Обратная связь — HTTP JSON + Authorization Bearer:
  GET  /api/lead-credentials?leadId=
  POST /api/webde-login-result
  POST /api/script-event
  POST /api/webde-wait-password  (long poll)
  POST /api/webde-login-grid-step
  GET  /api/webde-push-resend-poll?leadId=
  POST /api/webde-push-resend-result
  POST /api/webde-login-slot-done
  GET  /api/lead-klein-flow-poll?leadId=
  (+ клиентские POST вроде klein-anmelden-seen по сценарию)

Klein: spawn klein_simulation_api.py — те же server-url, lead-id, token.

Ручной старт из админки: POST /api/webde-login-start — тот же lead_simulation_api.py.

Файлы артефактов:
  login/cookies/*.json — куки сессий (выдача /api/lead-cookies, экспорт)
  data/webde-locks/*.lock — блокировки по email

Прочие Python-процессы из Node/планировщика:
  webde_layout_healthcheck.py — периодически из webdeLayoutHealthScheduler (env WEBDE_LAYOUT_HEALTH_INTERVAL_MS)
  webde_probe_batch.py (+ worker) — пакетный probe отпечатков из server (spawn)
  zip через execSync в админских config upload/zip-process (не webde_login)
